## 🔴 P0 严重 — 死锁 / 线程安全

### 1. QoS 引擎死锁 — 锁顺序反转

**文件:** `broker/qos_engine.go:315` + `broker/broker.go:679`

`doRetry()` 持有 `q.mu.Lock()` 后调用 `republish()` 回调:
```
doRetry() → q.mu.Lock()
  → republish() → topics.Match() → tt.mu.RLock()
                → deliverToClient() → b.mu.RLock()
                                    → cs.wmu.Lock()
```

同时 `disconnect()` 走相反顺序:
```
disconnect() → b.mu.Lock()
  → q.RemoveClient() → q.mu.Lock()  ← 死锁!
```

线程 A: `q.mu.Lock()` → 等待 `b.mu.RLock()` (被 B 的 `b.mu.Lock()` 阻塞)
线程 B: `b.mu.Lock()` → 等待 `q.mu.Lock()` (被 A 的 `q.mu.Lock()` 阻塞)

**修复:** 在 `doRetry()` 中解锁后再调用 republish，或收集待重试消息后批量处理。

### 2. client.go `nextPacketID()` 无锁保护

**文件:** `client/client.go:496-503`

`nextPacketID()` 直接读写 `c.nextPID` 无任何锁保护，被 `Publish()`、`Subscribe()`、`Unsubscribe()` 并发调用。在单 goroutine 使用场景下安全，但多 goroutine 并发发送时会产生重复 PacketID。

**修复:** 将 `nextPID` 改为 `atomic.Uint32` 或加锁保护。

### 3. Session.Save() 丢失关键字段

**文件:** `broker/session.go:246-276`

`Save()` 只持久化 `ClientID`、`IsClean`、`Subscriptions`、`Inflight`。恢复后的 Session 丢失:
- `Username` (默认空 → 授权失败)
- `KeepAlive` (默认 0 → 无心跳检测)
- `ProtocolVer` (默认 0 → MQTT 版本未知)

**修复:** 扩展 `SessionData` 结构体包含这些字段，并在 `Restore()` 中恢复。

## 🟡 P1 高 — 安全隐患 / 功能缺陷

### 4. Health HTTP Server 缺少超时

**文件:** `api/api.go:256`

```go
b.healthSrv = &http.Server{Handler: mux}
```

未设置 `ReadTimeout`、`WriteTimeout`、`IdleTimeout`，易受 Slowloris 攻击。

**修复:** 添加超时配置:
```go
b.healthSrv = &http.Server{
    Handler:      mux,
    ReadTimeout:  5 * time.Second,
    WriteTimeout: 5 * time.Second,
    IdleTimeout:  30 * time.Second,
}
```

### 5. Redis retained_store 的 MQTT `#` 转 Redis `*` 匹配过于宽泛

**文件:** `store/redis/retained_store.go:110-118`

Redis SCAN 的 glob `*` 匹配任意字符包括 `/`，而 MQTT `#` 只能匹配 topic 级别边界。虽然 `MatchRetained` 在第 95 行有二次 MQTT 匹配校验，但 Redis SCAN 仍会扫描不相关的 key，造成性能问题。

**修复:** 去掉 `#`→`*` 的转换（Reduced key precision），改为 `topic/#` → SCAN `topic/*`，但仅在 `#` 位于末尾时如此替换。

### 6. `writePacket` 和 `writePacketTo` 完全重复

**文件:** `broker/broker.go:546-575`

两个函数实现完全相同。`writePacket` 标注为 "read loop 专用"，`writePacketTo` 标注为 "通用"，但实现无差别。

**修复:** 删除 `writePacketTo`，统一使用 `writePacket`。

### 7. MQTT 5.0 ConnAck Properties 编码不完整

**文件:** `protocol/connect.go:339-347`

`encodeConnAck` 仅在 MQTT 5.0 模式下正确编码 Property Length = 0。对于 MQTT 3.1.1 的 CONNACK，如果 `pkt.Properties != nil` (意外情况)，会错误地编码 properties。该路径不应在生产中触发，但增加防御性编程。

**修复:** 遵循 codec protocolVersion 检查，MQTT 3.1.1 忽略 properties。

### 8. QoS 重试循环持有全局锁期间执行网络 I/O

**文件:** `broker/qos_engine.go:314-341`

`doRetry()` 持有 `q.mu.Lock()` 期间遍历所有 inflight 消息并调用 `republish()` (网络发送)。当 inflight 消息数量大 (max 100 per client × many clients) 时，全局 QoS 锁可能被持有数秒，阻塞所有 QoS 操作。

**修复:** 先收集需要重试的消息（快照），释放锁后再逐个 retry。

## 🟢 P2 中等 — 代码质量 / 可维护性

### 9. `sendConnAckRaw` 每次调用创建新 Codec

**文件:** `broker/broker.go:604`

认证失败时 `sendConnAckRaw` 创建 `protocol.NewCodec(...)` 仅发送一个 CONNACK。虽然这个路径不频繁，但可以复用 codec 实例。

### 10. server.go `conns` 与 broker.go `connections` 重复追踪

**文件:** `broker/server.go:31` + `broker/broker.go:42`

两个 map 都追踪活跃连接，语义重叠：
- `server.conns` — 用于 Stop 时关闭所有连接
- `broker.connections` — 用于 clientID → conn 映射

Stop 时 broker 已经遍历 `b.connections` 关闭连接 (broker.go:241-243)，server 的 `conns` map 成为冗余。

### 11. `DecConnections()` 实现误导

**文件:** `pkg/metrics/prometheus.go:145`

```go
func (m *prometheusMetrics) DecConnections() { m.disconnections.Inc() }
```

`DecConnections` 实际是递增 `disconnections` counter 而非递减 gauge。命名与实现不符。

**修复:** 重命名为 `OnDisconnect()` 或改为操作 gauge。

### 12. 密码 `string()` 转换在堆上分配不可清零内存

**文件:** `broker/broker.go:157`

`string(connectPkt.Password)` 在堆上创建新字符串，原 `[]byte` 可能被 GC 但新字符串无法清零。虽然认证实现使用了 `ConstantTimeCompare`，但如果未来有人添加日志，密码可能泄露。

**修复:** 在文档中明确标注 `Authenticate` 接口不应记录密码参数。

### 13. `cmd/main.go` 硬编码版本号

**文件:** `cmd/main.go:44`

```go
fmt.Printf("Shark-MQTT Broker v1.0.0 - listening on %s\n\n", cfg.ListenAddr)
```

版本号应通过 `-ldflags` 构建时注入而非硬编码。

### 14. Config 结构体有 `toml` 标签但不支持 TOML

**文件:** `config/config.go:13-49` + `config/loader.go:57`

所有配置字段都有 `toml` 标签但 `loadFile()` 明确拒绝 `.toml` 扩展名。

### 15. `server.conns` 在 accept 时添加但 connection handler 错误时可能不清理

**文件:** `broker/server.go:159-161`

连接被添加到 `server.conns`，defer 中会从 `server.conns` 中删除 (line 168)。但如果 broker 的 `HandleConnection` 内部也管理了连接生命周期 (`broker.connections`)，两个 map 的清理时序不同步。

## 🔵 P3 低 — 增强 / 优化

### 16. `replaceHash` / `replacePlus` 字符串拼接低效

**文件:** `store/redis/retained_store.go:120-143`

逐字符拼接应使用 `strings.ReplaceAll`:
```go
strings.ReplaceAll(strings.ReplaceAll(pattern, "#", "*"), "+", "[^/]")
```

### 17. 缺少 `var _ ConnectionHandler = (*Broker)(nil)` 接口检查

`broker.go` 中 Broker 实现了 `ConnectionHandler` 但无编译时检查。

### 18. 无 `-version` CLI flag

**文件:** `cmd/main.go:22-28`

添加 `-version` 标志打印版本信息后退出。

### 19. 格式一致性: 40 个文件被 `go fmt` 修改

运行 `go fmt ./...` 后有 40 个文件产生变更（主要是 LF/CRLF 换行符问题）。

### 20. `decodeProperties` 对未知 Property ID 返回错误而非静默跳过

**文件:** `protocol/properties.go:295-298`

对于 `propTypeMap` 中不存在的 property ID，`skipPropertyValue` 返回 error 导致整个 CONNECT 失败。MQTT 5.0 spec §1.5.5 允许对此类错误关闭连接，但也允许静默跳过。当前行为偏激进。
