## 本轮发现缺陷

### FIX-20260506-1: Retained message 指标计数覆盖/删除时漂移

**位置**: `broker/broker.go`
**严重度**: 中
**类型**: 可观测性 / 运维指标正确性

`Broker.handlePublish` 在处理 retained message 时，仅根据写入或删除动作对 `retainedCount` 做 `+1/-1`：

- 同一 topic 覆盖 retained message 会重复 `+1`，指标高于真实 retained topic 数。
- 删除不存在的 retained topic 会 `-1`，存在计数变负风险。
- 并发覆盖同一 topic 时，读写与计数更新没有统一临界区，仍可能漂移。

**影响**: Prometheus `retained_messages` gauge 不能反映真实 retained message 数量，容量告警、运行状态判断和问题排查会被误导。

**确认测试**:

- 新增 `TestBroker_RetainedMetricsCountUpdatesOnOverwriteAndDelete`
- 修复前结果：`retained metric updates = [1 2 1 0], want [1 1 0 0]`

**修复计划**:

1. 在 retained 写入/删除前检查 topic 是否已存在。
2. 将存在性检查、store 写入/删除、计数更新置于 broker 级 retained 临界区。
3. 覆盖时保持计数不变，删除不存在 topic 时保持计数不变。
4. 用定向测试和全量测试验证。

### FIX-20260506-2: MQTT 固定报头 flags 未做协议校验

**位置**: `protocol/codec.go`, `protocol/publish.go`, `protocol/subscribe.go`
**严重度**: 高
**类型**: MQTT 3.1.1 / 5.0 协议合规性

`Codec.Decode` 解析固定报头后直接进入各报文解码路径，没有验证固定报头 flags：

- `PINGREQ` 等非 PUBLISH 报文携带 QoS/Retain bit 时仍会被接受。
- `PUBLISH` 使用保留的 QoS=3 时仍会被接受。
- 这类畸形报文按 MQTT 规范应视为 malformed packet / protocol error。

**影响**: Broker 对非法客户端报文容忍度过高，可能导致互操作行为不符合规范，也会削弱协议层防御能力。

**确认测试**:

- 新增 `TestDecodeRejectsInvalidFixedHeaderFlags`
- 修复前 `PINGREQ with QoS bit set` 与 `PUBLISH with QoS 3` 均被错误解码成功。

**修复计划**:

1. 在 `Decode` 读取固定报头后立即校验 flags。
2. `PUBLISH` 禁止 QoS=3。
3. `PUBREL`、`SUBSCRIBE`、`UNSUBSCRIBE` 要求 QoS/Retain 低位组合为 `0x02`。
4. 其他非 PUBLISH 报文要求 QoS/Retain 低位组合为 `0x00`。
5. 保留 `DUP` 位的重传语义，不因 DUP=1 误杀合法重传。
6. 编码端为 `PUBREL`、`SUBSCRIBE`、`UNSUBSCRIBE` 自动设置规范要求的 QoS=1 / Retain=false，避免调用方遗漏固定 flags。

### FIX-20260506-3: CONNECT Will flag 组合约束缺失

**位置**: `protocol/connect.go`
**严重度**: 中
**类型**: MQTT 3.1.1 / 5.0 协议合规性

`ValidateConnect` 已校验 reserved bit、协议版本、ClientID 和部分 Will 规则，但缺少 MQTT CONNECT flags 的互斥约束：

- 当 `WillFlag=false` 时，`WillQoS` 必须为 `0`。
- 当 `WillFlag=false` 时，`WillRetain` 必须为 `false`。

**影响**: 畸形 CONNECT 可绕过协议校验，Broker 会接受不符合规范的客户端连接。

**确认测试**:

- 新增 `TestValidateConnectRejectsWillFlagsWithoutWillFlag`
- 修复前 `WillRetain=true` 或 `WillQoS=1` 且 `WillFlag=false` 均被错误接受。

**修复计划**:

1. 在 `ValidateConnect` 中补充 `WillFlag=false` 时的 Will QoS/Retain 约束。
2. 保持已有 `WillFlag=true` 时 Will QoS 范围校验。
3. 用定向测试和 protocol 包全量测试验证。
