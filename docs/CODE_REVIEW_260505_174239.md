## 项目概览

Shark-MQTT 是一个高性能 MQTT Broker，采用分层架构：

```
cmd/main.go → api.Broker → broker.MQTTServer + broker.Broker → protocol (codec)
                              +→ store (memory/redis/badger)
                              +→ plugin (hook system)
                              +→ pkg/logger + pkg/metrics
```

**测试覆盖**: 353 个测试（207 单元 + 77 集成 + 69 基准）

## 缺陷清单

### P0 — 关键缺陷

#### FIX-1: 会话过期清理缺失

- **位置**: `broker/broker.go`, `broker/session.go`
- **严重度**: 高 — 内存/存储泄漏
- **描述**: `Session.ExpiryInterval` 设置后，`Save()` 未计算 `ExpiryTime` 字段（`store/types.go:16` 已定义但从未赋值）。Broker 无清理协程，非干净会话断开后永不过期。
- **影响**: 长期运行的 Broker 会累积大量过期会话，消耗内存和存储空间。

#### FIX-2: Broker 端 QoS 2 重复检测缺失

- **位置**: `broker/broker.go:396-489` (`handlePublish`)
- **严重度**: 高 — 可能导致消息重复投递
- **描述**: 客户端 `client.go:456-471` 已实现 QoS 2 去重，但 Broker 端 `handlePublish` 不检查 DUP 标志，无 PacketID 去重映射。当 PUBREL 丢失且发布者重发 PUBLISH(DUP=1) 时，Broker 会重新处理消息。
- **影响**: 网络不稳定时 QoS 2 消息可能被重复投递给订阅者。

#### FIX-3: 指标采集未完整接入

- **位置**: `broker/broker.go`, `broker/topic_tree.go`
- **严重度**: 中 — 可观测性缺失
- **描述**: 5 个指标 setter（`SetOnlineSessions`, `SetOfflineSessions`, `SetRetainedMessages`, `SetSubscriptions`, `IncErrors`）有完整 Prometheus/noop 实现，但从未在生产代码中调用。
- **影响**: 运维无法监控会话数、订阅数、保留消息数等关键指标。

### P1 — 中等缺陷

#### FIX-4: Session.Save 持锁深拷贝

- **位置**: `broker/session.go:269-303`
- **严重度**: 中 — 并发性能
- **描述**: `Save()` 持有 `s.mu.RLock()` 期间深拷贝所有 inflight payload。大量 QoS 2 消息时阻塞所有会话读操作（如 `MatchesSubscription`）。
- **影响**: 高负载下写入延迟增加。

#### FIX-5: 客户端错误静默丢弃

- **位置**: `client/client.go:454-508`
- **严重度**: 低 — 错误可观测性
- **描述**: `handlePublish` 中 PUBACK/PUBREC 发送使用 `_ = c.codec.Encode(...)` 丢弃所有编码错误。
- **影响**: 连接断开等错误无法被调用方感知。

#### FIX-6: 保留消息 O(n) 线性匹配

- **位置**: `store/memory/memory.go:227-241`
- **严重度**: 低 — 大量保留消息时性能退化
- **描述**: `MatchRetained` 遍历所有保留消息并逐一调用 `protocol.MatchTopic`。无索引结构。
- **影响**: 1000+ 保留消息时订阅响应变慢。

### P2 — 代码质量

#### FIX-7: MQTT 3.1/5.0 解码路径不一致

- **位置**: `protocol/subscribe.go`, `protocol/publish.go`
- **严重度**: 低 — 维护复杂度
- **描述**: MQTT 5.0 路径使用 buffer-first 模式（先读全部字节到缓冲区再解析），MQTT 3.1.1 路径直接从 reader 增量读取。两个路径维护成本不同。
- **影响**: 添加新特性时需要理解两套解析逻辑。
