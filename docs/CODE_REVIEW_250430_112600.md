## 三、Critical 级别问题

### 🔴 C-1: Silent property decode errors — 协议错误静默丢弃

**文件:** `protocol/subscribe.go:29-30, 149-152, 221-223, 309-311`  
**问题:** MQTT 5.0 属性解码失败时，代码执行 `reader.Reset(data)` 后静默回退，整个 packet 被错误解析。这在 subscribe/suback/unsubscribe/unsuback 四个函数中各出现一次。  
**影响:** 协议违规 — MQTT 5.0 规范要求属性解码失败应关闭连接。当前行为产生静默数据损坏。  
**修复:** 属性解码失败应返回错误并 close 连接。

### 🔴 C-2: Badger/Redis 与 Memory 存储行为不一致 — 保留消息空 payload 处理

**文件:** `store/badger/retained_store.go:45-58`, `store/redis/retained_store.go:45-56`  
**问题:** Memory 存储在收到空 payload 时删除保留消息（符合 MQTT 规范），但 Badger 和 Redis 存储仍将其序列化存储，导致客户端订阅时收到空 payload 消息。  
**影响:** 跨存储后端的语义不一致 — 不同配置下 MQTT 行为不同。  
**修复:** Badger 和 Redis 的 `SaveRetained` 在 `len(payload)==0` 时应调用 `DeleteRetained`。

### 🔴 C-3: 客户端 readLoop 连接读取竞态

**文件:** `client/client.go:407-415`  
**问题:** `readLoop` 在 `c.mu.Lock()` 下获取 `conn`，释放锁后再调用 `c.codec.Decode(conn)`。`Disconnect()` 并发调用 `conn.Close()` 和 `c.cancel()`，存在 TOCTOU 竞态：连接在两次操作之间被关闭和替换。  
**影响:** 数据竞态 — 连接可能被关闭时还在读取，或新连接的数据被旧 readLoop 消费。  
**修复:** 使用连接级别的 context 控制读取生命周期，或在读取时持有锁。

### 🔴 C-4: Redis retained MatchRetained topic 模式匹配 BUG

**文件:** `store/redis/retained_store.go:117`  
**问题:** `topicPatternToRedis` 将 MQTT `+` 通配符映射为 Redis glob `[^/]`，但 `[^/]` 只匹配**单个**字符，而 MQTT `+` 应匹配整个 topic level（多个字符）。  
**影响:** `MatchRetained` 在 Redis 后端会丢失匹配的保留消息。  
**修复:** 使用 `KEYS` 命令（带 `*` 通配符）或使用正确的 Redis pattern 语法。

## 四、High 级别问题

### 🟠 H-1: 客户端无 PINGREQ / keepalive 机制

**文件:** `client/client.go`（全局）  
**问题:** 客户端在 CONNECT 中发送 `KeepAlive`，但从未发送 PINGREQ 包。服务端在 1.5×keepalive 时间内没有收到包会断开连接。  
**影响:** 所有使用此客户端的连接在生产环境中静默丢失。

### 🟠 H-2: 客户端 readLoop 无读取超时

**文件:** `client/client.go:415`  
**问题:** `c.codec.Decode(conn)` 调用未设置读取截止时间。如果服务端停止发送数据但不关闭连接，客户端 goroutine 永久阻塞。  
**影响:** goroutine 泄漏 + 客户端无法感知断开。

### 🟠 H-3: 客户端未检查 packet ID 是否已在 inflight 中使用

**文件:** `client/client.go:505-516`  
**问题:** `nextPacketID()` 使用 `atomic.CompareAndSwap` 循环 1-65535，但不检查返回的 ID 是否仍在 `inflight` 或 `pending` map 中使用。  
**影响:** packet ID 冲突导致 QoS 1/2 消息被错误确认。

### 🟠 H-4: 客户端 connect 失败时 WaitGroup 积累

**文件:** `client/client.go:155`  
**问题:** 每次 `Connect()` 调用 `c.wg.Add(1)`，但若连接失败（CONNACK 错误或网络错误），`readLoop` 可能不启动而 `wg.Done()` 未被调用。重新 Connect 时 WaitGroup 累积。  
**影响:** `Disconnect()` 中的 `c.wg.Wait()` 可能永久阻塞。

### 🟠 H-5: 服务端 Accept 循环无退避

**文件:** `broker/server.go:140-148`  
**问题:** `listener.Accept()` 返回永久性错误（如 `net.ErrClosed`）时，循环无退避地 `continue`，CPU 100% 自旋。  
**影响:** 监听器关闭条件（context cancellation）到达前，CPU 爆炸。

### 🟠 H-6: 服务端未使用的 option 字段

**文件:** `broker/options_server.go:15-25`  
**问题:** `writeBufferSize`、`readBufferSize`、`keepAlive`、`writeQueueSize`、`connectTimeout` 在 options 中定义但 server.go 从未应用于 TCP 连接。  
**影响:** 用户设置这些选项不会产生任何效果 — 静默失效。

### 🟠 H-7: 服务端/代理双重 max connections 检查

**文件:** `broker/server.go:152` + `broker/broker.go:111-119`  
**问题:** 服务端（从 `config.Config.MaxConnections`）和代理（从 `brokerOptions.maxConnections`）各自检查连接上限，使用不同的配置源。  
**影响:** 双重限制导致配置混淆。

### 🟠 H-8: MQTT 5.0 属性字段无值范围验证

**文件:** `protocol/properties.go:329, 373, 393`  
**问题:** `PayloadFormat`（应为 0/1）、`RequestProblemInfo`（应为 0/1）、`MaximumQoS`（应为 0/1）无范围验证。  
**影响:** 客户端可能发送非法值绕过协议检查。

### 🟠 H-9: Topic 过滤器不检查空 topic levels

**文件:** `protocol/topic.go:49-72`  
**问题:** `ValidateTopicFilter` 不检查连续 `/` 或尾随 `/`（如 `"a//b"`），产生空 topic levels，违反 MQTT 规范。  
**影响:** 非法的 topic 过滤器被接受和存储。

### 🟠 H-10: 服务端 session 中 NextPacketID 可能死循环

**文件:** `broker/session.go:227-240`  
**问题:** `NextPacketID` 在持有 `s.mu.Lock()` 的情况下循环最多 65535 次查找未使用的 packet ID。在极端条件下（全部 ID 都在使用中）这会长时间持有写锁。  
**影响:** 所有对 session 的并发操作被阻塞。

### 🟠 H-11: Prometheus 重复注册导致 panic

**文件:** `pkg/metrics/prometheus.go:41-131`  
**问题:** 每个 `NewPrometheusMetrics()` 调用使用 `reg.MustRegister()`，重复调用 panic。  
**影响:** 进程崩溃 — 测试中创建两个 metrics 实例或服务重启时触发。

**影响:** 生产环境中的 plugin panic 难以调试。

**文件:** `plugin/manager.go:53-60`  
**问题:** 不检查重复 plugin 名称或空字符串。  
**影响:** 相同名称的 plugin 在 `RegisteredPlugins()` 中被静默隐藏。

### 🟡 M-4: api.NewBroker 在配置验证失败后继续

**文件:** `api/api.go:129-131`  
**问题:** `cfg.Validate()` 错误仅记录日志，broker 仍然使用无效配置创建。  
**影响:** 错误在 `Start()` 时才暴露，难以排查。

### 🟡 M-5: 服务端 Stop 无超时

**文件:** `api/api.go:209-210`  
**问题:** `b.srv.Stop()` 和 `b.broker.Stop()` 无超时机制。  
**影响:** 关闭过程可能无限阻塞。

### 🟡 M-6: 服务端 ServerOptions writeBufferSize 等从未使用

**文件:** `broker/options_server.go` + `broker/server.go`  
**问题:** `writeBufferSize`, `readBufferSize`, `keepAlive` 定义了但未应用于连接。  
**影响:** 性能调优选项为死代码。

### 🟡 M-7: TopicTree 取消订阅后不清理孤儿节点

**文件:** `broker/topic_tree.go:108-125`  
**问题:** `unsubscribeNode` 仅在节点无订阅者且无子节点时删除，但不会递归向上清理。  
**影响:** 长期运行的内存泄漏。

### 🟡 M-8: 客户端 deliverResponse channel 满时静默丢弃

**文件:** `client/client.go:497-500`  
**问题:** `select { case ch <- pkt: default: }` 在 channel 满时静默丢弃响应包。  
**影响:** QoS 1/2 发布可能永久阻塞等待响应。

### 🟡 M-9: 客户端 inflightEntry.timer 从未使用

**文件:** `client/client.go:43-46, 185`  
**问题:** `timer` 字段从未被初始化或启动。QoS 1/2 消息无重传机制。  
**影响:** 网络丢包时 QoS 消息永久丢失。

### 🟡 M-10: 客户端 readLoop 不记录连接断开的错误

**文件:** `client/client.go:416-429`  
**问题:** `Decode` 错误仅用于触发清理，不记录日志。  
**影响:** 用户无法获知连接断开原因。

### 🟡 M-11: MQTT 3.1.1 Will TopicFlag 冗余

**文件:** `protocol/packets.go:155`  
**问题:** `WillTopicFlag` 始终等于 `WillFlag`，冗余字段。  
**影响:** 代码混淆。

### 🟡 M-12: codec 包大小检查 off-by-one

**文件:** `protocol/codec.go:39`  
**问题:** `fh.RemainingLength+5 > c.maxPacketSize` 假定 fixed header 总是 5 字节。实际为 2-5 字节。  
**影响:** 固定头小于 5 字节时允许略超限的包通过。

### 🟡 M-13: validateUTF8 不检查 MQTT 禁止的控制字符

**文件:** `protocol/codec.go:206-216`  
**问题:** 仅检查 null 字符，不检查 U+0001-U+001F、U+007F（禁止的控制字符）和 U+FEFF（BOM）。  
**影响:** MQTT 协议违规未被检测。

### 🟡 M-14: 客户端 OnMessage 回调内部锁排序可能导致死锁

**文件:** `client/client.go:449-458`  
**问题:** `handlePublish` 持有 `c.msgMu.RLock()` 调用用户回调 `fn()`。回调内调用 `Publish()`（需 `c.mu`）和另一个 goroutine 调用 `SetOnMessage()`（需 `c.msgMu.Lock()`）且持有 `c.mu` 时，产生锁排序死锁。  
**影响:** 特定调用模式下的死锁。

### 🟡 M-15: Memory 存储 session 无 TTL/过期处理

**文件:** `store/memory/memory.go`  
**问题:** `SessionData` 包含 `ExpiryTime` 但从未检查或清理过期 session。  
**影响:** 内存无限增长。

### 🟡 M-16: Badger MessageStore ListMessages 静默跳过错误

**文件:** `store/badger/message_store.go:99-105`  
**问题:** `item.ValueCopy(nil)` 和 `json.Unmarshal` 错误被静默 `continue`，导致结果不完整。  
**影响:** 静默数据丢失。

**问题:** 全局 `var idCounter int64` 意味着所有实例共享计数器。
