### HIGH-1: Property string reader lacks UTF-8 validation

**File:** `protocol/properties.go:452-470` (`readStringFromReader`)

**Issue:** The main `readString()` in `codec.go:180-198` validates UTF-8 correctness and rejects null characters (U+0000), per MQTT spec §1.5.3. However, `readStringFromReader()` used for MQTT 5.0 property strings does NOT perform any validation. This means ContentType, ResponseTopic, AssignedClientID, ReasonString, AuthMethod, and UserProperty key/value strings can contain invalid UTF-8 or embedded nulls.

**Impact:** Non-conformant MQTT 5.0 clients could send malformed UTF-8 in properties, causing downstream issues in logging, storage, or API consumers.

**Fix:** Add UTF-8 validation to `readStringFromReader()` matching the logic in `readString()`.

```go
// After reading the string bytes, add:
if err := validateUTF8(s); err != nil {
    return "", err
}
```

### HIGH-2: Session restore from persistent store never called on reconnect

**File:** `broker/broker.go:168` (`HandleConnection`)

**Issue:** When a client reconnects with CleanSession=false, the code checks `b.sessions.SessionExists()` which only looks at the in-memory map. If the broker was restarted, all persisted sessions (in Redis/Badger) are orphaned — they exist in the store but are never restored. The `Manager.Restore()` method exists but is never invoked from the connection handler.

**Impact:** After broker restart, clients reconnecting with CleanSession=false:
- Lose all previous subscriptions
- Don't receive queued offline messages
- Get a fresh session (effectively CleanSession=true behavior)

**Fix:** In `HandleConnection`, before calling `SessionExists()`, check the session store if no in-memory session exists:

```go
isResuming := b.sessions.SessionExists(connectPkt.ClientID)
if !isResuming && !connectPkt.Flags.CleanSession && b.sessionStore != nil {
    if exists, _ := b.sessionStore.IsSessionExists(ctx, connectPkt.ClientID); exists {
        restored, err := b.sessions.Restore(ctx, connectPkt.ClientID)
        if err == nil && restored != nil {
            isResuming = true
            // Re-register subscriptions in TopicTree
            for topic, qos := range restored.Subscriptions {
                b.topics.Subscribe(topic, connectPkt.ClientID, qos)
            }
        }
    }
}
```

### MEDIUM-1: Integration test helper doesn't override MetricsAddr

**File:** `tests/integration/retained_test.go:15-28` (`testBrokerWithRetain`)

**Issue:** This helper creates a broker with `cfg.ListenAddr = ":0"` but doesn't set `cfg.MetricsAddr = ":0"`, so the health server tries to bind to the default `:9090`. When `:9090` is occupied or multiple retained tests run, it logs a warning. Six tests are affected.

**Fix:** Add `cfg.MetricsAddr = ":0"` after `cfg.ListenAddr = ":0"`.

---

### MEDIUM-2: Session resume discards new CONNECT parameters

**File:** `broker/session.go:83-123` (`CreateSession`)

**Issue:** When an existing in-memory session is resumed (`exists && !CleanSession && isResuming`), the old session object is returned as-is. The new CONNECT packet's KeepAlive, Username, and ProtocolVersion are silently ignored. Per MQTT 3.1.1 §3.1.2.10 and MQTT 5.0 §3.1.2.11, the server should use the new KeepAlive value.

**Impact:** If a client changes its KeepAlive between connections, the old value persists. If credentials change but the session is reused, the old username stays in session metadata.

**Fix:** Update session fields from the new CONNECT packet when resuming:

```go
if exists && !connectPkt.Flags.CleanSession && isResuming {
    existing.KeepAlive = connectPkt.KeepAlive
    existing.Username = connectPkt.Username
    existing.ProtocolVer = connectPkt.ProtocolVersion
    existing.LastActivity = time.Now()
    return existing
}
```

### LOW-1: Client hardcodes MQTT 3.1.1, no MQTT 5.0 support

**File:** `client/client.go:100` (`Connect`)

**Issue:** The client always sets `ProtocolVersion: protocol.Version311`. There's no option to use MQTT 5.0 features (properties, reason codes, session expiry, etc.).

**Impact:** The client library can only talk MQTT 3.1.1. Any broker requiring MQTT 5.0 would reject connections.

**Fix:** Add a `WithProtocolVersion(v uint8)` option to client options. Not blocking — the built-in client is primarily for smoke testing.

---

### LOW-2: `encodeConnect` mutates caller's packet FixedHeader

**File:** `protocol/connect.go:277` (`encodeConnect`)

**Issue:** `encodeConnect` sets `pkt.FixedHeader.PacketType = PacketTypeConnect` and `pkt.RemainingLength = buf.Len()` on the caller's packet. If a caller encodes the same packet object twice, the second call's RemainingLength starts from the first call's value rather than being recalculated. Currently harmless since RemainingLength is always overwritten, but surprising mutation.

**Fix:** Acceptable as defensive code (the previous CONNECT bug motivated this). Low priority.

---

### LOW-3: `AckPubRec` double-lock with TOCTOU on callback pointer

**File:** `broker/qos_engine.go:219-237` (`AckPubRec`)

**Issue:** `sendPubRel` callback is read under `RLock`, then the lock is released and re-acquired as `Lock` to update state. Between these two operations, `sendPubRel` could theoretically be changed by `SetCallbacks()`. In practice, callbacks are set once at broker startup.

**Impact:** Negligible. The callback pointer would only change during teardown, and `AckPubRec` wouldn't be called then.

---

### LOW-4: `WillHandler.Stop()` clears maps while holding lock, then waits on goroutines

**File:** `broker/will_handler.go:40-50` (`Stop`)

**Issue:** `Stop()` acquires `wh.mu`, clears `wills` and `cancel` maps, then calls `wh.wg.Wait()` while still holding the lock. Delayed will goroutines from `TriggerWill` only access `wh.mu` during startup (to register the cancel func), not during execution, so no deadlock. However, clearing the cancel map and then waiting is correct since the goroutines' ctx is cancelled, causing them to exit promptly.

**Impact:** None. Pattern is safe in current code.
