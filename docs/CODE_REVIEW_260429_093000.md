## 2. 严重缺陷（Critical）

### C-001: Session Takeover 竞态条件导致连接丢失

**文件**: `broker/broker.go:191-198`, `broker/broker.go:279-295`

**描述**: 当具有相同 ClientID 的新连接到达时，broker 关闭旧连接并将新连接注册到 `connections` 映射。但旧连接的 `readLoop` goroutine 检测到连接关闭后，调用 `abnormalDisconnect` → `disconnect`，而 `disconnect` 在第 293 行执行 `delete(b.connections, clientID)`——这会删除**新**连接的条目，导致新连接后续写入失败。

**复现流程**:
1. conn1 (clientID="A") 已连接
2. conn2 (clientID="A") 到达
3. `HandleConnection` 中关闭 conn1，注册 conn2 到 `connections["A"]`
4. 发送 CONNACK 给 conn2 ✓
5. conn1 的 readLoop 检测到 close，调用 `abnormalDisconnect` → `disconnect`
6. `disconnect` 执行 `delete(b.connections, "A")` —— 删除了 conn2！
7. conn2 的后续消息写入失败（`writePacket` 中找不到连接）

**影响**: 导致 `TestPersistentSession_KickPrevious` 测试失败，生产环境中会导致 session takeover 后新连接无法正常通信。

**修复方案**: 在 `HandleConnection` 注册新连接后，在关闭旧连接之前设置一个标记，或在 `disconnect` 中检查当前连接是否与被踢连接相同：

```go
// broker.go HandleConnection, line 191-198
b.mu.Lock()
if old, exists := b.connections[clientID]; exists {
    old.conn.Close()
    // 不在这里 delete，让旧 readLoop cleanup 自己处理
    // 但需要确保 disconnect 不会删除新连接
}
b.connections[clientID] = &clientState{conn: conn, codec: c}
b.mu.Unlock()
```

更安全的方案是在 `disconnect` 中验证要删除的连接条目就是自己的连接：

```go
// disconnect 中
b.mu.Lock()
if cs, exists := b.connections[clientID]; exists && cs.conn == conn {
    delete(b.connections, clientID)
}
b.mu.Unlock()
```

但 `disconnect` 函数没有 `conn` 参数，需要重构。

### C-002: 内存存储 GetSession 浅拷贝导致潜在数据竞态

**文件**: `store/memory/memory.go:46-56`

**描述**: `GetSession` 方法对 `SessionData` 执行浅拷贝 `copied := *data`，但 `data.Inflight` 是指向 `store.InflightMessage` 的指针映射。虽然外部调用者（`session.go:Restore`）进行了深度拷贝，但接口契约没有强制要求，任何直接调用 `GetSession` 的代码都可能修改内部状态。

**影响**: 潜在的数据竞态和内存安全问题。

**修复方案**: 在 `GetSession` 中完全深度拷贝 `Inflight` 字段。

### C-003: 默认 AllowAllAuth 在 cmd/main.go 中硬编码

**文件**: `cmd/main.go:48-51`

**描述**: 主入口点总是使用 `AllowAllAuth`，虽然有 stderr 警告，但没有机制阻止在生产环境中运行。相当于一个开放代理，允许任何人发布/订阅任意主题。

**影响**: 严重安全隐患。如果部署到公网，任何人都可以无认证连接、发布任意消息。

**修复方案**:
1. 要求显式设置认证提供者
2. 添加 `--auth` 命令行选项
3. 或在检测到非交互式终端时拒绝启动 AllowAllAuth

## 3. 中等缺陷（Medium）

### M-001: MQTT 5.0 Session Expiry Interval 未实现

**文件**: `broker/session.go`, `broker/broker.go`

**描述**: CONNECT 包中的 `SessionExpiryInterval` 属性被正确解码（`protocol/properties.go:171-176`），但在 broker 层完全被忽略。会话使用硬编码的 24h 过期（`config.go:66`），不遵守客户端请求的过期时间。

### M-002: 离线消息队列缺失

**文件**: `broker/broker.go:421-438`

**描述**: 当 CleanSession=false 的客户端断开连接时，发送到其订阅主题的消息会丢失。MQTT 规范要求 broker 在客户端重新连接后投递这些消息。当前实现没有离线消息队列机制，`messageStore` 在整个代码中从未被使用。

### M-003: UNSUBSCRIBE 主题未验证

**文件**: `broker/broker.go:476-488`

**描述**: `handleUnsubscribe` 直接处理主题而不验证。根据 MQTT 3.1.1 §3.10.3，UNSUBSCRIBE 中的主题过滤器应该是有效的（但不允许通配符）。当前实现允许客户端取消订阅无效的主题过滤器。

### M-004: CONNACK 缺少 MQTT 5.0 能力属性

**文件**: `broker/broker.go:582-610`

**描述**: MQTT 5.0 broker 应在 CONNACK 中返回能力属性（ReceiveMaximum、MaximumQoS、RetainAvailable、WildcardSubAvailable 等）。当前 `sendConnAck` 不包含任何属性，标准的 MQTT 5.0 客户端可能行为异常。

### M-005: StaticAuth ACL 默认拒绝行为不一致

**文件**: `broker/auth.go:80-109`

**描述**: `StaticAuth.CanPublish` 和 `CanSubscribe` 对未知用户返回 `false`（拒绝），这是正确的默认拒绝行为。但 `AllowAllAuth` 作为授权器时返回 `true`。当同时使用 `StaticAuth` 作为授权器和 `AllowAllAuth` 作为认证器时，所有发布/订阅都会被拒绝，这不符合预期。

### M-006: TopicTree 匹配无缓存

**文件**: `broker/topic_tree.go:78-88`

**描述**: 每次 `Match` 调用都从根节点进行完整的递归遍历。对于大量订阅者或深度嵌套的主题，性能可能下降。没有订阅匹配结果的缓存。

### M-007: Plugin Dispatch 在首个错误时停止

**文件**: `plugin/manager.go:64-85`

**描述**: Dispatch 在遇到第一个插件错误时立即返回，不执行后续插件。这使得多个独立插件不可能同时运行，其中一个插件失败会阻止其他插件执行。

## 4. 轻微缺陷（Minor）

### L-001: errs 包未被使用

**文件**: `errs/errors.go`

**描述**: 整个 `errs` 包定义了 15 个错误变量，但项目中没有任何文件导入该包（除了测试）。错误在 `broker/auth.go` 和 `store/errors.go` 中重复定义。

### L-002: 重复错误定义

`ErrAuthFailed` 同时在 `errs/errors.go:21` 和 `broker/auth.go:14` 定义。两者语义相同但位于不同的 sentinel 值。

### L-003: writePacket 错误被静默忽略

**文件**: `broker/broker.go:567-580`

当 `writePacket` 失败时（如连接已关闭），错误仅记录到调试日志。对于 QoS 1/2 消息，这意味着发布者可能已确认但订阅者从未收到消息。调用者不知道投递失败。

### L-004: RemainingLength 未验证上限

**文件**: `protocol/codec.go:33`

代码检查 `fh.RemainingLength+5 > c.maxPacketSize`，但 RemainingLength 被解码为 int，理论上可达 268MB（4 字节变长编码）。对于 32 位系统，大的 RemainingLength 可能导致内存分配问题。

### L-005: client.go 中 connect/disconnect 缺少连接检查

**文件**: `client/client.go:68-158`

`Connect` 方法中的 `c.connected` 检查和 `c.mu` 锁之间存在 TOCTOU 竞态窗口。虽然影响很小，但在极端并发场景下可能导致双重连接。

### L-006: 基准测试报告 [no tests to run]

**文件**: `tests/bench/`

`go test` 运行 `tests/bench` 包时报告 `[no tests to run]`。基准函数（`Benchmark*`）在 `go test` 默认模式下不运行，需要 `-bench=.` 标志。这不影响功能但 CI 输出混乱。

### L-007: 测试中硬编码超时可能不稳定

多个集成测试使用硬编码的 `2 * time.Second` 超时。在负载高的 CI 环境中可能导致不稳定的测试失败。

### L-008: 缺少协议模糊测试

`protocol/codec.go` 中的 `Decode` 函数处理网络输入但没有模糊测试。MIME 类型的输入可能导致 panic 或安全问题。

## 7. 架构评估

### 优点
- 清晰的分层架构：protocol → broker → api
- 良好的接口抽象（Authenticator, Authorizer, SessionStore）
- 函数选项模式（Option pattern）使用规范
- TopicTree 实现正确，支持 $SYS 系统主题保护
- MQTT 5.0 属性编解码完整
- 竞态检测通过

### 待改进
- Broker.HandleConnection 过于庞大（~130 行），应拆分为多个职责清晰的函数
- session 管理逻辑分布在 broker.go 和 session.go 中，职责边界模糊
- 缺乏中间件/拦截器机制，认证/授权/日志混在业务逻辑中
- WriteQueueSize 配置项定义了但未在任何地方使用
