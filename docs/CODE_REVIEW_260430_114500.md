## Remaining Unfixed Issues (Newly Discovered)

### H-001: Data Race — `NextPacketID` uses RLock but mutates state

**File:** `broker/session.go:223-243`
**Severity:** High
**Root Cause:** `NextPacketID()` acquires `s.mu.RLock()` (read lock) but writes to `s.packetIDSeq` at lines 229, 231, 239, 241.

```go
func (s *Session) NextPacketID() uint16 {
    s.mu.RLock()        // READ lock
    defer s.mu.RUnlock()
    for attempts := 0; attempts < 65535; attempts++ {
        id := s.packetIDSeq
        s.packetIDSeq++  // WRITE under read lock = DATA RACE
        ...
    }
}
```

Go's memory model requires exclusive access for writes. Multiple goroutines calling `NextPacketID()` concurrently (e.g., QoS 1/2 delivery from broker to subscriber) will race on `s.packetIDSeq`. This would be flagged by `go test -race`. Under high concurrency, the sequence can produce duplicate or skipped packet IDs.

**Fix:** Change `RLock`/`RUnlock` to `Lock`/`Unlock`.

### H-002: QoS Engine `maxInflight` configured but never enforced

**File:** `broker/qos_engine.go:78,165-206`
**Severity:** High
**Root Cause:** `QoSEngine.maxInflight` is stored in the struct (default 100) but neither `TrackQoS1` nor `TrackQoS2` checks the per-client inflight count before inserting.

```go
func (q *QoSEngine) TrackQoS1(clientID, packetID, topic, payload, retain) {
    q.mu.Lock()
    defer q.mu.Unlock()
    // Missing: if len(q.inflight[clientID]) >= q.maxInflight { return error }
    q.inflight[clientID][packetID] = &InflightMessage{...}
}
```

The broker advertises `ReceiveMaximum` in CONNACK properties but never enforces the limit. A slow or malicious subscriber can accumulate unlimited inflight messages, consuming unbounded memory.

**Fix:** Add inflight limit check before insertion in both `TrackQoS1` and `TrackQoS2`. Also propagate the check to `broker.go` to reject QoS 1/2 PUBLISH when the subscriber's inflight is full.

### M-001: Config validation errors silently ignored — broker starts with invalid config

**File:** `api/api.go:129-131`
**Severity:** Medium
**Root Cause:** `NewBroker` calls `cfg.Validate()` but only logs errors via `log.Printf`. The broker continues to start, deferring the failure to later (possibly in production).

```go
if err := o.cfg.Validate(); err != nil {
    log.Printf("[api] config validation warning: %v", err)  // logged, not returned
}
```

**Impact:** Invalid storage backend, missing Redis address, bad TLS config — all silently accepted.

**Fix:** Return the validation error. Update `NewBroker` signature or add a separate `Validate()` method on the API Broker.

### M-002: `decodeUnsubscribe` silently returns partial results on malformed topic

**File:** `protocol/subscribe.go:225-231`
**Severity:** Medium
**Root Cause:** In MQTT 5.0 UNSUBSCRIBE decoding, errors from `readStringFromReader` cause `break` instead of `return nil, err`.

```go
for reader.Len() > 0 {
    topic, err := readStringFromReader(reader)
    if err != nil {
        break  // should return the error
    }
    topics = append(topics, topic)
}
```

If a topic string contains invalid UTF-8, the broker processes a partial unsubscribe list, leaving some topics still subscribed.

**Fix:** `if err != nil { return nil, err }`

### M-003: Session store error silently discarded during existence check

**File:** `broker/broker.go:170`
**Severity:** Medium
**Root Cause:** The error from `sessionStore.IsSessionExists()` is discarded with `_`.

```go
if exists, _ := b.sessionStore.IsSessionExists(ctx, connectPkt.ClientID); exists {
```

When the store has a transient error (e.g., Redis network blip), `exists=false`, and the broker creates a fresh session instead of restoring the persistent one. The client loses all previous subscriptions and inflight messages.

**Fix:** Log the error at minimum. Optionally attempt `Restore` anyway if the existence check errors.

### M-004: Client `nextPacketID` infinite spin on PID exhaustion

**File:** `client/client.go:505-515`
**Severity:** Medium
**Root Cause:** `nextPacketID` uses a tight `CompareAndSwap` spin loop that never checks whether the returned PID is actually available in the inflight map.

```go
func (c *MQTTClient) nextPacketID() uint16 {
    for {  // loops forever if all 65535 PIDs in use
        old := c.nextPID.Load()
        next := old + 1
        if next > 65535 { next = 1 }
        if c.nextPID.CompareAndSwap(old, next) {
            return uint16(old)
        }
    }
}
```

**Impact:** Under extreme concurrent publish load, the client livelocks at 100% CPU.

**Fix:** Add a max-attempt guard (e.g., 65535 attempts) and return an error if exhausted.

---

### M-005: No duplicate PacketID detection for received QoS 2 PUBLISH (client)

**File:** `client/client.go:452-489`
**Severity:** Medium
**Root Cause:** `handlePublish` for QoS 2 does not track received PacketIDs. Per MQTT 3.1.1 §4.3.3 and MQTT 5.0 §4.3.3, the receiver MUST remember the PacketID of a QoS 2 PUBLISH until the flow completes, to detect and discard retransmissions.

**Impact:** If the broker retransmits a QoS 2 PUBLISH (e.g., PUBREC was lost), the client delivers the message to `onMessage` twice.

**Fix:** Track received QoS 2 PacketIDs in a map and discard duplicates until PUBCOMP is sent.

---

### M-006: MaximumQoS validated in wrong packet type (dead code)

**File:** `protocol/connect.go:55-57`
**Severity:** Low-Medium
**Root Cause:** `decodeConnect` checks `props.MaximumQoS` — but MaximumQoS is a CONNACK property sent from server to client, not a CONNECT property sent from client to server.

```go
if props != nil && props.MaximumQoS != nil && *props.MaximumQoS > 1 {
    return nil, ErrMalformedPacket  // dead code — clients never send this
}
```

**Fix:** Remove the dead check from `decodeConnect`. If MaximumQoS validation is desired, place it in the CONNACK encoding path.
