## Critical Findings

### C-1: Race condition on Codec.protocolVersion (protocol/codec.go:14)

`protocolVersion` is a mutable field on `Codec` struct, set during `decodeConnect` and `encodeConnect`. Multiple goroutines can call `Decode`/`Encode` concurrently on the same `Codec` instance. No synchronization around `protocolVersion` read/write. Each connection needs its own Codec instance, or the field must be protected by a mutex.

## High-Severity Findings

### H-1: Session restore does not populate Subscriptions map (broker/broker.go:169-187)

When restoring a session from persistent store, subscriptions are added to `TopicTree` via `b.topics.Subscribe()` but never added to `Session.Subscriptions` map. `deliverToClient()` calls `sess.MatchesSubscription()` which checks this map — returning false, so restored subscribers never receive messages. **This is a correctness bug: persistent sessions silently fail.**

### H-2: Session save/disconnect race (broker/broker.go:313-319)

`disconnect()` saves session to store, then removes from in-memory map. Between save and remove, a concurrent `CreateSession` for the same clientID could save a new session. The old session's save could overwrite the new session's data in the store. Requires atomic save+remove under lock.

### H-3: nil auth/authorizer/pluginManager causes panic (api/api.go:126-130, api/api.go:135)

`NewBroker` passes nil values to `broker.WithAuth(nil)`, `broker.WithAuthorizer(nil)`, `broker.WithPluginManager(nil)` when options are not set. If the broker does not nil-check before calling methods on these interfaces, it panics. Similarly, `WithConfig(nil)` causes nil-pointer dereference on field access.

### H-4: Unbounded Prometheus label cardinality (pkg/metrics/prometheus.go:146-188)

`IncRejections(reason string)`, `IncMessagesDropped(reason string)`, `IncErrors(component string)` use arbitrary strings as Prometheus label values. If these strings come from user input, this creates unbounded cardinality, leading to memory exhaustion and process crash.

### H-5: No keepalive/ping mechanism in client (client/client.go)

Client sends KeepAlive in CONNECT but never sends PINGREQ. After broker keepalive timeout (typically 1.5x), connection is silently dropped. Client doesn't notice until next read/write.

### H-6: readLoop has no read deadline (client/client.go:403-411)

`readLoop` calls `c.codec.Decode(conn)` with no read deadline. If broker stops sending data without closing the connection, the goroutine blocks indefinitely. Combined with H-5, this creates a permanent goroutine leak.

### H-7: IPv6 address parsing broken (client/options.go:28-37)

`WithAddr` uses manual character-by-character port parsing that doesn't handle IPv6 addresses (`[::1]:1883`). Should use `net.SplitHostPort`.

### H-8: QoS 2 publish reuses same channel for PUBREC and PUBCOMP (client/client.go:222-241)

A single `respCh` channel (buffer size 1) is used for both PUBREC and PUBCOMP responses. If PUBCOMP arrives before the first select reads PUBREC, the type assertion fails and returns a false error.

### H-9: Config never validated (multiple callers)

`cfg.Validate()` is defined but never called by `cmd/main.go`, `api/api.go`, or `api.Run()`. Invalid configs (negative max packet size, empty listen addr) silently cause runtime failures.

## Medium-Severity Findings

### M-1: Duplicate sentinel errors (protocol/errors.go and errs/errors.go)
`ErrInvalidPacket`, `ErrPacketTooLarge`, `ErrUnsupportedVersion` defined in both packages. Code using `errors.Is()` from one layer won't match errors from the other.

### M-2: Property decode errors silently discarded (protocol/subscribe.go:27-30,149-152,220-223,308-311)
When `decodeProperties` fails in decodeSubscribe, decodeSubAck, decodeUnsubscribe, decodeUnsubAck, the reader is reset and parsing continues. Per MQTT 5.0 spec, malformed properties should cause connection close.

### M-3: Memory store returns shallow copies (store/memory/memory.go:123-132,144-157,215-226)
`GetMessage`, `ListMessages`, `MatchRetained` copy the struct but share the Payload `[]byte` backing array with the store. Callers can corrupt store state by modifying returned payloads.

### M-4: Redis GET errors silently ignored (store/redis/retained_store.go:89-100, message_store.go:92-101)
During SCAN+GET loops, transient Redis errors from GET are silently discarded via `continue`. Caller gets incomplete results with no error indication.

### M-5: Will topic not validated for wildcards (broker/broker.go:220-226, will_handler.go:774-790)
Will topic is stored and later published without validation. MQTT spec prohibits wildcards in publish topics. `ValidatePublishTopic` is only called in `handlePublish`, not on will messages.

### M-6: NextPacketID can return ID already in Inflight map (broker/session.go:223-232)
Monotonically increments and wraps at 65535 without checking if the generated ID is already in use in the Inflight map. Could cause message corruption for QoS 1/2 delivery.

### M-7: Old will goroutine not cancelled on new will registration (broker/will_handler.go:58-104)
When a client reconnects and registers a new will, the old will's delayed goroutine still fires. Old and new wills can both be published for the same disconnect.

### M-8: Timing side-channel in password comparison (broker/auth.go:76, auth_file.go:95)
`subtle.ConstantTimeCompare([]byte(password), []byte(expected))` uses user-supplied password length as first argument. Comparison time varies with password length, leaking it.

### M-9: Retained store errors silently ignored (broker/broker.go:431,435,562-569)
`DeleteRetained`, `SaveRetained`, `MatchRetained` errors discarded without logging or metrics.

### M-10: Data race on Session.KeepAlive/Username (broker/session.go:94-102, broker/broker.go:241,353)
Session.KeepAlive and Username written under session lock in CreateSession, but read without lock in broker.HandleConnection and readLoop.

### M-11: Plugin dispatch panic has no stack trace (plugin/manager.go:73-77)
`recover()` captures panic value but doesn't include goroutine stack trace. Debugging plugin crashes is nearly impossible.

### M-12: Redis store missing prefix guard (store/redis/message_store.go, retained_store.go)
`ListMessages` and `MatchRetained` don't check for empty keyPrefix (unlike `ListSessions` which does). Empty prefix causes full Redis DB scan.

### M-13: Badger store doesn't set Timestamp on RetainedMessage (store/badger/retained_store.go:45-57)
`SaveRetained` creates `RetainedMessage` without setting `Timestamp`. Memory store correctly uses `time.Now()`. Inconsistent behavior.

### M-14: IsConnected uses exclusive lock (client/client.go:387-391)
Uses `c.mu.Lock()` (write lock) for read-only check of `c.connected`. Should use `c.mu.RLock()`.

### M-15: Plugin Register doesn't validate name uniqueness or emptiness (plugin/manager.go:53-60)
No check for duplicate plugin names or empty name string. Multiple plugins with same name get silently hidden in RegisteredPlugins.

### M-16: Dispatch doesn't nil-check data parameter (plugin/manager.go:65)
Nil `*Context` passed to plugins without validation. Each plugin must handle nil individually.

### M-17: log.SetFlags in init() mutates global state (pkg/logger/slog.go:9-11)
Package init function modifies global `log` package flags, affecting all packages in the binary.

### M-18: inflightEntry.timer never used - dead feature (client/client.go:43-46,185)
Struct has timer field but it's never initialized. Inflight messages from QoS 1/2 publishes never get retried. Entries leak until disconnect.

### M-19: accept loop spins on permanent errors (broker/server.go:140-149)
Permanent Accept errors (like ErrClosed) are treated same as temporary errors. Loop spins with no backoff on fatal errors.

### M-20: encodeAuth always writes reason code byte (protocol/subscribe.go:455)
MQTT 5.0 spec says AUTH reason code is optional. Always writing it could cause interop issues with strict parsers.

### M-21: QoS subscription value not validated (client/client.go:266)
TopicSubscription.QoS (byte, 0-255) passed directly to protocol.TopicFilter.QoS. Value 3 (reserved) not rejected.

### M-22: Config.Validate not called from any path (config/config.go:74)
Validate method exists but is never invoked by cmd, api, or any caller.
